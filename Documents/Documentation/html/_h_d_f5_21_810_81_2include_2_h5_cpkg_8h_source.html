56 /****************************************************************************
57  *
58  * We maintain doubly linked lists of instances of H5C_cache_entry_t for a
59  * variety of reasons -- protected list, LRU list, and the clean and dirty
60  * LRU lists at present. The following macros support linking and unlinking
61  * of instances of H5C_cache_entry_t by both their regular and auxilary next
62  * and previous pointers.
63  *
64  * The size and length fields are also maintained.
65  *
66  * Note that the relevant pair of prev and next pointers are presumed to be
67  * NULL on entry in the insertion macros.
68  *
69  * Finally, observe that the sanity checking macros evaluate to the empty
70  * string when H5C_DO_SANITY_CHECKS is FALSE. They also contain calls
71  * to the HGOTO_ERROR macro, which may not be appropriate in all cases.
72  * If so, we will need versions of the insertion and deletion macros which
73  * do not reference the sanity checking macros.
74  * JRM - 5/5/04
75  *
76  * Changes:
77  *
78  * - Removed the line:
79  *
80  * ( ( (Size) == (entry_ptr)->size ) && ( (len) != 1 ) ) ||
81  *
82  * from the H5C__DLL_PRE_REMOVE_SC macro. With the addition of the
83  * epoch markers used in the age out based cache size reduction algorithm,
84  * this invariant need not hold, as the epoch markers are of size 0.
85  *
86  * One could argue that I should have given the epoch markers a positive
87  * size, but this would break the index_size = LRU_list_size + pl_size
88  * + pel_size invariant.
89  *
90  * Alternatively, I could pass the current decr_mode in to the macro,
91  * and just skip the check whenever epoch markers may be in use.
92  *
93  * However, any size errors should be caught when the cache is flushed
94  * and destroyed. Until we are tracking such an error, this should be
95  * good enough.
96  * JRM - 12/9/04
97  *
98  *
99  * - In the H5C__DLL_PRE_INSERT_SC macro, replaced the lines:
100  *
101  * ( ( (len) == 1 ) &&
102  * ( ( (head_ptr) != (tail_ptr) ) || ( (Size) <= 0 ) ||
103  * ( (head_ptr) == NULL ) || ( (head_ptr)->size != (Size) )
104  * )
105  * ) ||
106  *
107  * with:
108  *
109  * ( ( (len) == 1 ) &&
110  * ( ( (head_ptr) != (tail_ptr) ) ||
111  * ( (head_ptr) == NULL ) || ( (head_ptr)->size != (Size) )
112  * )
113  * ) ||
114  *
115  * Epoch markers have size 0, so we can now have a non-empty list with
116  * zero size. Hence the "( (Size) <= 0 )" clause cause false failures
117  * in the sanity check. Since "Size" is typically a size_t, it can't
118  * take on negative values, and thus the revised clause "( (Size) < 0 )"
119  * caused compiler warnings.
120  * JRM - 12/22/04
121  *
122  * - In the H5C__DLL_SC macro, replaced the lines:
123  *
124  * ( ( (len) == 1 ) &&
125  * ( ( (head_ptr) != (tail_ptr) ) || ( (cache_ptr)->size <= 0 ) ||

3455 /****************************************************************************
3456  *
3457  * structure H5C_t
3458  *
3459  * Catchall structure for all variables specific to an instance of the cache.
3460  *
3461  * While the individual fields of the structure are discussed below, the
3462  * following overview may be helpful.
3463  *
3464  * Entries in the cache are stored in an instance of H5TB_TREE, indexed on
3465  * the entry's disk address. While the H5TB_TREE is less efficient than
3466  * hash table, it keeps the entries in address sorted order. As flushes
3467  * in parallel mode are more efficient if they are issued in increasing
3468  * address order, this is a significant benefit. Also the H5TB_TREE code
3469  * was readily available, which reduced development time.
3470  *
3471  * While the cache was designed with multiple replacement policies in mind,
3472  * at present only a modified form of LRU is supported.
3473  *
3474  * JRM - 4/26/04
3475  *
3476  * Profiling has indicated that searches in the instance of H5TB_TREE are
3477  * too expensive. To deal with this issue, I have augmented the cache
3478  * with a hash table in which all entries will be stored. Given the
3479  * advantages of flushing entries in increasing address order, the TBBT
3480  * is retained, but only dirty entries are stored in it. At least for

4046  * Automatic cache size adjustment:
4047  *
4048  * While the default cache size is adequate for most cases, we can run into
4049  * cases where the default is too small. Ideally, we will let the user
4050  * adjust the cache size as required. However, this is not possible in all
4051  * cases. Thus I have added automatic cache size adjustment code.
4052  *
4053  * The configuration for the automatic cache size adjustment is stored in
4054  * the structure described below:
4055  *
4056  * size_increase_possible: Depending on the configuration data given
4057  * in the resize_ctl field, it may or may not be possible
4058  * to increase the size of the cache. Rather than test for
4059  * all the ways this can happen, we simply set this flag when
4060  * we receive a new configuration.
4061  *
4062  * flash_size_increase_possible: Depending on the configuration data given
4063  * in the resize_ctl field, it may or may not be possible
4064  * for a flash size increase to occur. We set this flag
4065  * whenever we receive a new configuration so as to avoid
4066  * repeated calculations.
4067  *
4068  * flash_size_increase_threshold: If a flash cache size increase is possible,
4069  * this field is used to store the minimum size of a new entry
4070  * or size increase needed to trigger a flash cache size
4071  * increase. Note that this field must be updated whenever
4072  * the size of the cache is changed.
4073  *
4074  * size_decrease_possible: Depending on the configuration data given
4075  * in the resize_ctl field, it may or may not be possible
4076  * to decrease the size of the cache. Rather than test for
4077  * all the ways this can happen, we simply set this flag when
4078  * we receive a new configuration.
4079  *
4080  * resize_enabled: This is another convenience flag which is set whenever
4081  * a new set of values for resize_ctl are provided. Very
4082  * simply,
4083  *
4084  * resize_enabled = size_increase_possible ||
4085  * size_decrease_possible;
4086  *
4087  * cache_full: Boolean flag used to keep track of whether the cache is
4088  * full, so we can refrain from increasing the size of a
4089  * cache which hasn't used up the space allotted to it.
4090  *
4091  * The field is initialized to FALSE, and then set to TRUE
4092  * whenever we attempt to make space in the cache.
4093  *
4094  * size_decreased: Boolean flag set to TRUE whenever the maximum cache
4095  * size is decreased. The flag triggers a call to
4096  * H5C__make_space_in_cache() on the next call to H5C_protect().
4097  *
4098  * resize_in_progress: As the metadata cache has become re-entrant, it is
4099  * possible that a protect may trigger a call to
4100  * H5C__auto_adjust_cache_size(), which may trigger a flush,
4101  * which may trigger a protect, which will result in another
4102  * call to H5C__auto_adjust_cache_size().
4103  *
4104  * The resize_in_progress boolean flag is used to detect this,
4105  * and to prevent the infinite recursion that would otherwise
4106  * occur.
4107  *
4108  * Note that this issue is not hypothetical -- this field
4109  * was added 12/29/15 to fix a bug exposed in the testing
4110  * of changes to the file driver info superblock extension
4111  * management code needed to support rings.
4112  *
4113  * msic_in_progress: As the metadata cache has become re-entrant, and as
4114  * the free space manager code has become more tightly
4115  * integrated with the metadata cache, it is possible that
4116  * a call to H5C_insert_entry() may trigger a call to
4117  * H5C_make_space_in_cache(), which, via H5C__flush_single_entry()
4118  * and client callbacks, may trigger an infinite regression
4119  * of calls to H5C_make_space_in_cache().
4120  *
4121  * The msic_in_progress boolean flag is used to detect this,
4122  * and prevent the infinite regression that would otherwise
4123  * occur.
4124  *
4125  * Note that this is issue is not hypothetical -- this field
4126  * was added 2/16/17 to address this issue when it was
4127  * exposed by modifications to test/fheap.c to cause it to
4128  * use paged allocation.
4129  *
4130  * resize_ctl: Instance of H5C_auto_size_ctl_t containing configuration
4131  * data for automatic cache resizing.
4132  *
4133  * epoch_markers_active: Integer field containing the number of epoch
4134  * markers currently in use in the LRU list. This value
4135  * must be in the range [0, H5C__MAX_EPOCH_MARKERS - 1].
4136  *
4137  * epoch_marker_active: Array of boolean of length H5C__MAX_EPOCH_MARKERS.
4138  * This array is used to track which epoch markers are currently
4139  * in use.
4140  *
4141  * epoch_marker_ringbuf: Array of int of length H5C__MAX_EPOCH_MARKERS + 1.
4142  *
4143  * To manage the epoch marker cache entries, it is necessary
4144  * to track their order in the LRU list. This is done with
4145  * epoch_marker_ringbuf. When markers are inserted at the
4146  * head of the LRU list, the index of the marker in the
4147  * epoch_markers array is inserted at the tail of the ring
4148  * buffer. When it becomes the epoch_marker_active'th marker

4166  * epochs. These entries (if any) can then be evicted and
4167  * the cache size reduced -- ideally without evicting any

4171  * set will be accessed before the limit.

4174  * the index or slist. While they are of type
4175  * H5C__EPOCH_MARKER_TYPE, and have associated class

4181  * to the constant structure epoch_marker_class defined
